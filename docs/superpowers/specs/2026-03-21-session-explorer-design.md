# Session Explorer — Design Spec

## Summary

Add a session explorer dashboard as the landing page of Unravel, replacing the current upload-first flow. SREs can browse past analyses, preview session details in a slide-out panel, re-read reports and chat transcripts, and start new analyses — turning Unravel from a single-use tool into a workspace.

## Motivation

The current app flow is ephemeral: upload → analyze → chat → gone. SREs working across multiple support bundles and incidents have no way to reference past analyses, compare findings, or return to a previous investigation. The assignment doc explicitly encourages exploring "what else can you do within the problem domain that would be interesting" — session persistence and a browsable history demonstrate product thinking beyond the single-analysis happy path.

## Golden Requirements Impact

- **GR-6a (new addendum)**: Analysis results, generated reports, extracted bundle metadata, and LLM observability data may be persisted to local JSON storage. Raw bundle archives are still cleaned up after extraction. No data is sent to external services beyond the LLM provider.
- All other golden requirements remain unaffected.

## Architecture

### Data Flow

```
┌─────────────────────────────────────────────────────┐
│              Analysis Pipeline (existing)            │
│  Upload → Extract → Analyze → Stream Report → Chat  │
└──────────────────────┬──────────────────────────────┘
                       │ on completion
                       ▼
┌──────────────────────────────────────────────────────┐
│              Session Persistence Layer (new)          │
│                                                      │
│  SessionPersistence class                             │
│  ├── save_session(session_data) → writes JSON        │
│  ├── list_sessions() → reads index                   │
│  ├── get_session(id) → loads full report + chat      │
│  ├── update_notes(id, notes) → edits user metadata   │
│  └── delete_session(id) → removes session data       │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────────┐
│           /app/data/sessions/ (Docker volume)        │
│                                                      │
│  sessions.json          ← lightweight index          │
│  {session_id}/                                       │
│    ├── report.json      ← full analysis report       │
│    └── chat.json        ← chat transcript (if any)   │
└──────────────────────────────────────────────────────┘
```

### New API Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/sessions` | List all sessions (returns index data) |
| `GET` | `/api/sessions/{id}` | Get full session detail (report + chat) |
| `PATCH` | `/api/sessions/{id}` | Update notes/tags |
| `DELETE` | `/api/sessions/{id}` | Remove session and its data |

### Storage Structure

**Index file** (`sessions.json`): Array of session summaries, loaded for the dashboard table. Kept lightweight for fast reads.

```json
[
  {
    "id": "uuid",
    "bundle_name": "support-bundle-prod.tar.gz",
    "file_size": 14872000,
    "timestamp": "2026-03-21T15:30:00Z",
    "status": "completed",  // values: "completed" | "error"
    "cluster": "prod-east-1",
    "namespaces": ["default", "kube-system", "app"],
    "k8s_version": "v1.28.4",
    "node_count": 3,
    "findings_summary": [
      {"severity": "critical", "title": "CrashLoopBackOff in app/api-server"},
      {"severity": "warning", "title": "Resource limits not set on 3 deployments"}
    ],
    "llm_meta": {
      "provider": "anthropic",
      "model": "claude-sonnet-4-20250514",
      "input_tokens": 12500,
      "output_tokens": 3200,
      "latency_ms": 8400
    },
    "eval_score": 0.85,
    "notes": "",
    "tags": []
  }
]
```

**Per-session directory** (`{session_id}/`):
- `report.json` — Full analysis report (the complete structured report as generated by the pipeline)
- `chat.json` — Array of chat messages (role, content, timestamp) if the SRE used chat. Read-only transcript, not resumable. When viewing a saved session, the existing `ChatPhase` component renders the transcript in read-only mode (messages displayed, input disabled).

### Docker Volume

Add to `docker-compose.yml`:

```yaml
services:
  backend:
    volumes:
      - session-data:/app/data/sessions

volumes:
  session-data:
```

Data persists across `docker compose down`. Cleared with `docker compose down -v`.

## Frontend Design

### Dashboard (Landing Page)

Replaces the current upload screen as the app's entry point.

**Layout:**
- Header: "Unravel" branding + "New Analysis" button (top-right)
- Stats bar: Total sessions, sessions with critical findings, completed count, errored count
- Filter bar: Search (bundle name, cluster, notes), status dropdown, severity dropdown
- Session table: Dense rows with columns — Bundle (name + notes subtitle), Cluster, Findings (severity counts), Analyzed (relative time), Status

**Table behavior:**
- Sorted by timestamp descending (most recent first)
- Click a row → opens detail panel
- Hover highlight for scanability
- Empty state: "No analyses yet" with prominent "New Analysis" CTA

### Detail Panel (Slide-out)

Opens from the right when a table row is clicked. ~340px wide.

**Sections:**
1. **Header**: Bundle filename, analysis timestamp, file size, close button
2. **Notes** (editable): Freeform text field. SRE can add customer name, ticket number, context. Saves via `PATCH /api/sessions/{id}`
3. **Bundle Metadata** (auto-extracted): Cluster name, node count, namespaces, K8s version — pulled from the bundle during analysis
4. **Findings**: List of finding titles with severity dots (critical=red, warning=amber, info=blue)
5. **Actions**: "Open Full Report" button (navigates to report+chat view), delete button
6. **LLM Observability** (collapsible): Provider, model, token counts, latency, eval score

### Navigation Changes

The current app uses `useState<AppPhase>` for navigation (no router library). We extend this pattern rather than adding a router — the app is single-page with no need for deep-linking or URL-based navigation, and adding `react-router` for a prototype would be over-engineering.

**Extended AppPhase state machine:**

```
'explorer' → 'upload' → 'dashboard' (existing analysis flow)
'explorer' → 'dashboard' (viewing a saved session, loaded from JSON)
'dashboard' → 'explorer' (back button)
'upload' → 'explorer' (cancel)
```

- `AppPhase` gains a new `'explorer'` state (replaces `'upload'` as the initial state)
- The existing `'upload'` and `'dashboard'` phases remain unchanged
- When viewing a saved session, the app loads report/chat data from the API into the same state variables used by the existing `ReportPhase` and `ChatPhase` components
- "New Analysis" button sets phase to `'upload'`
- "Back to Explorer" sets phase to `'explorer'`
- After completing a new analysis → session auto-saved, user can continue to chat or return to explorer

## Backend Changes

### SessionPersistence Class (new, separate from existing SessionStore)

New module: `src/backend/app/sessions/persistence.py`

The existing `SessionStore` (`src/backend/app/sessions/store.py`) is an **in-memory store** that manages active session lifecycle — it holds `extracted_files`, `chroma_collection_name`, and other runtime data needed during analysis and chat. It stays unchanged.

The new `SessionPersistence` class is a **separate persistence layer** that saves derived data (reports, chat transcripts, metadata) to disk on analysis completion. The in-memory store manages the active session; `SessionPersistence` creates a durable record of the finished session.

**Relationship:** When an analysis completes, the pipeline calls `SessionPersistence.save_session()` to write the report, metadata, and findings to JSON. The in-memory `SessionStore` continues to manage the active session for chat. When the in-memory session expires/evicts (existing TTL logic), the persisted record remains on disk.

- File-based JSON storage (no database dependency)
- Atomic writes (write-to-temp + rename) — single-process asyncio, no file locking needed
- Methods: `save_session`, `list_sessions`, `get_session`, `update_session`, `delete_session`
- Configurable data directory via environment variable (`SESSION_DATA_DIR`, default `/app/data/sessions`)

### Pipeline Integration

Hook into the existing analysis pipeline to auto-save on completion:

1. After the analysis stream completes and the report is generated, call `SessionPersistence.save_session()` with:
   - Session ID, bundle filename, file size
   - Auto-extracted bundle metadata (cluster, namespaces, k8s version, node count)
   - Findings summary (severity + title for each finding)
   - LLM metadata from the tracker
   - Eval score
2. Write the full report to `{session_id}/report.json`
3. During chat, append each message pair to `{session_id}/chat.json`

### Metadata Extraction

Extract from the bundle during analysis (these are files the pipeline already reads):
- **Cluster name**: from `cluster-info/` or `cluster-resources/` paths
- **K8s version**: from `cluster-resources/server-version.json`
- **Node count**: from `cluster-resources/nodes.json`
- **Namespaces**: from namespace directories in the bundle structure

This extraction happens in the existing analysis pipeline — the data is available, we just need to capture and pass it to SessionPersistence.

### Session API Routes

New route module: `src/backend/app/api/session_routes.py`

- `GET /api/sessions` — Read and return `sessions.json`
- `GET /api/sessions/{id}` — Read `sessions.json` entry + load `{id}/report.json` and `{id}/chat.json`
- `PATCH /api/sessions/{id}` — Update notes/tags in `sessions.json` entry
- `DELETE /api/sessions/{id}` — Remove entry from `sessions.json` and delete `{id}/` directory

## What Does NOT Change

- The analysis pipeline logic (extract, chunk, analyze, RAG, eval)
- The report rendering (ReportPhase component)
- The chat functionality (ChatPhase component)
- The LLM logger (still emits to stdout as before)
- Provider configuration and failover logic

## Scope Boundaries

**In scope:**
- Session persistence (JSON files)
- Dashboard landing page with table view
- Detail panel with editable notes
- Bundle metadata extraction
- Read-only chat transcript viewing
- Session deletion

**Out of scope (future):**
- Resumable chat (requires context reconstruction)
- Session export (PDF, CSV)
- Session comparison (side-by-side findings)
- Search across report content (full-text search)
- Session sharing or multi-user access (GR-11: no auth/multi-tenancy)

## Testing Strategy

- **SessionPersistence**: Unit tests for CRUD operations, atomic write correctness (crash during write doesn't corrupt index), corrupt file handling
- **API endpoints**: Integration tests for each endpoint, error cases (missing session, invalid ID)
- **Frontend**: Component tests for SessionExplorer table rendering, detail panel interactions
- **Pipeline integration**: Integration test verifying a completed analysis produces a session entry
- **Docker volume**: Manual verification that data persists across container restarts
